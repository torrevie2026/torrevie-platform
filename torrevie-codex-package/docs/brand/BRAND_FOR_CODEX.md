# Brand for Codex — Design Tokens and Implementation Rules

This is the file `packages/ui` is built from. It translates `docs/brand/BRAND_STRATEGY_POSITIONING.md`, `VISUAL_IDENTITY.md`, and `BRAND_GUIDELINES.md` into values you can paste directly into code. If a screen's styling ever conflicts with this file, this file is wrong and needs to be fixed, not the screen worked around; flag it.

## Logo assets

Located at `assets/logo/`:

- `torrevie_logo_color.png` — full color mark, light backgrounds only (the Admin Portal top bar, the Customer Portal top bar, light-background emails, print/PDF exports).
- `torrevie_logo_white.png` — white reversed mark, dark backgrounds only (the login/auth screens' navy hero panel, any full-navy marketing surface).

Never use `torrevie_logo_color.png` on a navy or dark background. Never regenerate, recolor, or redraw the mark; these two files are the only approved logo assets until a corrected wordmark lockup is delivered separately.

## Color tokens

Use these exact values. Do not introduce any other color into the UI, including for status indicators, without deriving it from this set (see "Status colors" below for the one sanctioned extension).

### CSS custom properties (`packages/ui/tokens.css`)

```css
:root {
  --torrevie-navy: #162449;
  --torrevie-white: #ffffff;
  --torrevie-black: #0a0a0a;
  --torrevie-teal: #0d9488;
  --torrevie-steel-blue: #4a6fa5;
  --torrevie-light-grey: #f2f4f7;

  /* Semantic aliases — use these in component code, not the raw palette names,
     so a future token change only touches this file. */
  --color-bg-primary: var(--torrevie-white);
  --color-bg-surface: var(--torrevie-light-grey);
  --color-bg-inverse: var(--torrevie-navy);
  --color-text-primary: var(--torrevie-navy);
  --color-text-inverse: var(--torrevie-white);
  --color-text-muted: #5b6472; /* derived, near-black grey for secondary text, not in the core palette but acceptable for muted body copy */
  --color-accent: var(--torrevie-teal);
  --color-accent-secondary: var(--torrevie-steel-blue);
  --color-border: #e2e6ec; /* derived, light border tone, keeps light-grey surfaces from feeling flat */
}
```

### Status colors (the one sanctioned extension beyond the core palette)

The core palette has no red, amber, or green. Product UI needs status indication (error states, approval-pending states, success confirmations, work-order status, expense status). Use these, and only these, restricted to small indicators, badges, and inline messages, never as large background fields:

```css
--color-status-success: #1a7f5a;
--color-status-warning: #b8860b;
--color-status-error: #b3261e;
```

These are muted, not saturated, so they sit quietly next to navy and teal rather than competing with the "one accent color" rule in the Visual Identity document. Do not use pure red/green/amber from a default component library; override them to these values wherever a UI kit (for example shadcn/ui) ships its own defaults.

## Typography

- Font family: **Inter** (variable font) everywhere. Load via `next/font/google` or a self-hosted variable font file, not a CDN link with a flash-of-unstyled-text risk.
- Weights used: 400 (Regular, body text), 600 (Semibold, subheadings, labels), 700 (Bold, headings, the wordmark).
- No second font family anywhere in the product, including PDFs generated by the platform (proposal exports, invoices) and system emails.
- Arabic text uses Inter's Arabic character support where available; if Inter's Arabic coverage is insufficient at implementation time, flag it rather than silently substituting a different family — this is a genuine open question to resolve with Torrevie, not a decision to make unilaterally.

```css
:root {
  --font-family-base: 'Inter', system-ui, sans-serif;
}
body { font-family: var(--font-family-base); }
```

## Spacing, radius, and elevation

Not fully specified in the source brand documents; the following are Codex's implementation defaults, consistent with the "minimal, uncluttered" and "no decoration beyond the logo" direction in the Visual Identity document. Treat these as sensible defaults, not locked brand law like color and type:

- Border radius: small and consistent, 6–8px on cards, inputs, and buttons. No fully rounded ("pill") buttons except where a status badge specifically calls for it.
- Shadows: minimal to none. A card sits on a light-grey background or has a 1px `--color-border` outline, rather than relying on drop shadow for separation, matching "no drop shadows... beyond the logo itself" from the Visual Identity document.
- Spacing scale: an 8px base unit (8, 16, 24, 32, 48, 64), generous whitespace, consistent with "the majority of every layout should be white or near-white space."

## Recurring UI devices (from the Visual Identity graphic language)

- **Section divider:** a 2–3px teal (`--color-accent`) underline beneath a section heading. No full-width decorative bars.
- **Callout / info card:** `--color-bg-surface` background, no border, a 3–4px solid left edge in teal or navy depending on emphasis.
- **Comparison layout:** for any before/after or plan-comparison UI (for example, a subscription plan picker in the Admin Portal), use a two-column block, left column `--color-bg-surface`, right column `--color-bg-inverse` with white text, mirroring the "Comparison Slide" pattern in the PowerPoint template reference.
- **Data visualization:** flat bar/line charts using navy, teal, and steel blue only, no 3D chart effects, no shadowed data points, gridlines removed or minimal, value labels shown directly where practical (this mirrors the "Data Slide" pattern in the PowerPoint template reference and applies directly to CRM/FSM/TEX dashboard charts).
- **Hero / auth screens only:** large, soft, low-opacity navy circles as background decoration, exactly as used on the PowerPoint title and closing slides. Never used behind body text or dense data screens.

## Logo placement rules in the product

- Admin Portal and Customer Portal top navigation bar: `torrevie_logo_color.png` on a white or light-grey bar, paired with the "TORREVIE" wordmark in Inter Bold.
- Login/auth screen: navy full-bleed panel using `torrevie_logo_white.png`, matching the PowerPoint title-slide treatment (large low-opacity navy circles, white reversed mark, slogan in teal near the bottom).
- System-generated PDFs (proposals, invoices, exports): follow the Word Template layout already defined in Torrevie's brand package — header with the color mark and the slogan, footer with contact details and pagination.
- Emails: color mark on a white background in the header, "Optimize. Execute. Scale." in teal in the footer, alongside `torrevie.com` and `hello@torrevie.com`.

## Copy and microcopy rules (tone of voice applied to UI text)

Every button label, empty state, error message, onboarding step, and email subject line follows the tone-of-voice principles in `BRAND_STRATEGY_POSITIONING.md`:

- Short, declarative. "Opportunity created." not "Great news! Your opportunity has been successfully created!"
- No exclamation points anywhere in the product, including success toasts and onboarding.
- Specific over vague. An error message names what happened and what to do next, not "Something went wrong."
- No hype words: never disrupt, revolutionize, game-changing, seamless, effortless, supercharge.
- Empty states describe the method plainly: "No opportunities yet. Add your first account to get started." not "Your pipeline is waiting to be filled with amazing deals!"

## What Codex does if a brand question is not answered here

Stop and ask, rather than inventing a new color, a new pattern, or a second typeface under time pressure. Log the gap in `docs/architecture/PROGRESS.md` as an open branding question so it can be resolved once and folded back into this file, per the governance rule in `BRAND_GUIDELINES.md`.
